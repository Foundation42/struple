struple

Schema-free binary tuples whose bytes sort themselves.

Pack values into bytes that are directly memcmp-comparable — drop them into any byte-ordered store and they sort in the correct semantic order, no custom comparator. It's a stream, not a message, and a canonical one — which makes it a natural fit for databases and CRDTs alike. One wire format, six byte-identical implementations.

6 languages memcmp-orderable streamable canonical schema-free zero-dependency

The idea

A struple is a sequence of typed values packed into a byte buffer so that comparing the raw bytes gives you the right order:

std.mem.order(pack(a), pack(b))  ==  the semantic order of a and b

Every element starts with a one-byte type code, chosen so the type byte alone gives the cross-type order. Within a type, payloads are encoded to preserve order under memcmp: integers carry their width in the type code (negatives in excess form), floats use the IEEE-754 total-ordering transform, and variable-length values are 0x00-terminated so a shorter value sorts before a longer one that extends it. It's the FoundationDB tuple idea, rebuilt clean.

Read it as it streams a stream of fields, not a message

A struple isn't a blob you parse all at once — it's a sequence of self-delimiting elements. Each one carries its own length in its type code and framing, so there's no length prefix to wait for.

48 7573657200 212a ▸ 06

A reader walks the stream one element at a time, so you can decode field 1, field 2, … as the bytes arrive — pull out the few fields you need and stop, or skip the rest without decoding it. That's the opposite of a random-access format like Cap'n Proto, which needs the whole message resident to follow its pointers.

And because each element leads with its most-significant bytes (type code first, then big-endian payload), a prefix of the stream already orders correctly — perfect for database key scans, range queries, and tailing a log.

Built for CRDTs deterministic + ordered = mergeable

Two properties make struple an unusually good substrate for conflict-free replicated data (CRDTs) and content-addressed systems:

Canonical bytes

The same value always encodes to exactly the same bytes — map keys sorted, floats canonicalized, no whitespace or ordering ambiguity — and identically across all six languages. So you can hash, compare, dedup, and merge by raw bytes, and two replicas independently produce byte-identical encodings. Most formats (JSON, Protobuf, MessagePack) have no canonical form at all.

Ordered keys

Because keys are memcmp-ordered, you can mint a key between any two existing keys — fractional indexing for sequence CRDTs (RGA / LSEQ / Fugue-style), ordered maps and sets that merge by range, and operation logs that sort deterministically — for tuples of any typed values, not just strings.

Deterministic encoding + a total order is exactly the substrate that replicated, mergeable, ordered data is built on.

The wire format

Order falls straight out of the type byte:

nil < undefined < false < true < −int < 0 < +int < float32 < float64 < timestamp < uuid < string < bytes < array < map < set

0x01 nil
0x02 undefined
0x05/06 false / true
0x10–1F −int (1–16 bytes)
0x20 zero
0x21–30 +int (1–16 bytes)
0x0F / 31 big −/+ int (> i128)
0x34 / 35 float32 / 64
0x40 timestamp
0x44 uuid
0x48 / 49 string / bytes
0x50 array
0x52 / 54 map / set
